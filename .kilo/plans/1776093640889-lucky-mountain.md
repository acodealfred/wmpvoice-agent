# Plan: Fix Biometric Capturing During Survey Session

## Problem Analysis

### Current Behavior

1. **Blink Rate Change showing 0% and "Normal" stress**:
   - Blink rate is calculated in `useBiometrics.ts` (lines 401-405) based on baseline comparison
   - The baseline is captured during 30-second baseline session
   - However, the frontend sends biometric updates via `/biometrics` endpoint independently of question timing
   - When the LLM calls `record_survey_response` tool, it uses `_current_blink_rate_change` which may be 0 if baseline wasn't properly established

2. **Face Emotion showing duplicated values**:
   - `useVideoCapture` analyzes emotions every 3000ms (defined in `analyzeInterval`)
   - The last detected emotion is stored in `lastEmotion` state and used for all POST requests
   - There's no tracking of which emotion was dominant during each question period
   - Multiple frames with same emotion result in repeated identical values

### Root Cause

The core issue is **temporal correlation missing**: biometric data is sent continuously to backend but not associated with specific question/response time windows. When recording a survey response, the system uses whatever was last cached, not what was captured during the actual question answer period.

**Data Flow Problem**:
```
VideoPanel → useBiometrics (33ms) → POST /biometrics → Backend (_current_blink_rate_change)
                                         ↑
useVideoCapture (3000ms) ───────────────┘
                                         ↓
                              LLM calls record_survey_response
                              → Uses self._current_blink_rate_change (may be stale/0)
```

## Proposed Fix

### Solution: Time-Bucketed Biometric Tracking

Track biometric readings within time windows aligned with each question's response period, then compute statistics when recording the response.

#### Step 3: Compute Aggregated Biometrics When Recording Response

**File**: `app/frontend/src/App.tsx`

Instead of relying on cached backend values, compute aggregates client-side and send with survey response:

```typescript
const computeQuestionBiometrics = (questionId: string) => {
  const data = questionBiometrics[questionId];
  if (!data) return { blinkRateChange: 0, faceEmotion: 'NEUTRAL' };
  
  // Average blink rate change (excluding outliers)
  const blinkChanges = data.blinkRateChanges.filter(v => v !== 0);
  const avgBlinkRateChange = blinkChanges.length > 0
    ? blinkChanges.reduce((a, b) => a + b, 0) / blinkChanges.length
    : 0;
  
  // Most common emotion in the window
  const emotionCounts: Record<string, number> = {};
  data.emotions.forEach(e => {
    emotionCounts[e.emotion] = (emotionCounts[e.emotion] || 0) + 1;
  });
  const faceEmotion = Object.entries(emotionCounts)
    .sort((a, b) => b[1] - a[1])[0]?.[0] || 'NEUTRAL';
  
  return { blinkRateChange: avgBlinkRateChange, faceEmotion };
};
```

#### Step 4: Backend Enhancement for Rolling Averages (Recommended)

**File**: `app/backend/rtmt.py`

Instead of single-value storage, maintain rolling window buffers:

```python
class RTMiddleTier:
    # Add after _current_face_emotion declaration
    _blink_rate_history: list = []
    _face_emotion_history: list = []
    _MAX_HISTORY_SIZE = 50  # ~5 seconds at 10 updates/sec
    
    async def _update_biometric_history(self, blink_change: float, face_emotion: str):
        self._blink_rate_history.append(blink_change)
        self._face_emotion_history.append(face_emotion)
        
        if len(self._blink_rate_history) > self._MAX_HISTORY_SIZE:
            self._blink_rate_history.pop(0)
            self._face_emotion_history.pop(0)
    
    def _get_average_blink_rate_change(self) -> float:
        valid_changes = [c for c in self._blink_rate_history if c != 0]
        if not valid_changes:
            return 0.0
        return sum(valid_changes) / len(valid_changes)
    
    def _get_dominant_emotion(self) -> str:
        if not self._face_emotion_history:
            return 'NEUTRAL'
        # Count occurrences
        emotion_counts: dict = {}
        for emotion in self._face_emotion_history:
            emotion_counts[emotion] = emotion_counts.get(emotion, 0) + 1
        return max(emotion_counts, key=emotion_counts.get)
```

Update `update_biometrics` in `app.py` to call history update:
```python

    data = await request.json()
    # ... existing code ...
    await rtmt._update_biometric_history(blink_rate_change, face_emotion)
```

### Why This Works

1. **Blink Rate**: By maintaining a rolling window (last ~5 seconds), we capture the actual blink rate during question response period. Average of valid readings provides accurate representation.

2. **Face Emotion**: Tracking all emotions and selecting the dominant one ensures we capture the most frequent emotion during the question period, not just the last detected one.

3. **Temporal Alignment**: Both metrics are captured continuously and aggregated when needed, ensuring correlation with question timing without complex state management.
